## Part 1. Redis vs 메시지 큐 — 무엇을 결정하는가

이 선택은 "캐시 도구 고르기"가 아니라 **예매 요청을 동기로 처리할지, 비동기로 처리할지**를 가른다.
시스템 안에 이 결정이 걸리는 지점은 **두 곳**이다.

| 지점 | 역할 | 선택의 의미 |
|------|------|-------------|
| **(A) 입장 제어(Admission)** | 피크 트래픽으로부터 시스템 보호 (순번 비노출) | 동시 활성자 수 제한 vs 큐 버퍼링 |
| **(B) 예매 처리 파이프라인** | 좌석 점유 임계영역 실행 | 동기 락 처리 vs 비동기 직렬 처리 |

---

### 1.1 두 아키텍처 안

#### ▷ 안 A — Redis 중심 (동기 예매)
```
[Client]
  │ 1. 입장 토큰 요청
  ▼
[Redis: 활성자 카운터(상한 K)]  ── 초과 시 429/503 + Retry-After (순번 X)
  │ 2. 토큰 보유자만 예매 요청 (동기 HTTP)
  ▼
[API] ──▶ [Redis 분산 락(좌석 단위)] ──▶ [DB 상태전이 AVAILABLE→HELD]
  │ 3. 즉시 성공/실패 응답
  ▼
[Client] (결과 바로 수신)
```
- 예매 결과를 **요청-응답 한 번에** 받는다. UX 단순.
- 동시성은 **분산 락 + DB 낙관 락**으로 보장.

#### ▷ 안 B — 메시지 큐 중심 (비동기 예매)
```
[Client]
  │ 1. 예매 요청 → 즉시 "접수됨 + 추적ID" 응답 (202)
  ▼
[API] ──▶ [Message Queue: 예매 요청 스트림]
                    │  (파티션/순서 보장)
                    ▼
            [Consumer(들)] ── 좌석별 직렬 처리 ──▶ [DB 상태전이]
                    │
                    ▼
            [결과 저장] ◀── Client가 추적ID로 폴링/SSE로 확인 (2)
```
- 요청 수신과 처리를 **분리(decoupling)** → 스파이크를 큐가 흡수.
- 같은 좌석을 같은 파티션으로 보내면 **락 없이 직렬화**되어 경합 자체가 사라짐.
- 대신 결과가 **비동기** → 폴링/SSE UX, 운영 복잡도↑.

---

### 1.2 비교표 (트레이드오프)

| 기준 | 안 A (Redis 동기) | 안 B (MQ 비동기) |
|------|-------------------|------------------|
| 정합성 보장 방식 | 분산 락 + 낙관 락 | 파티션 직렬화 (락 불필요) |
| 피크 스파이크 흡수 | 활성자 수 제한으로 입장 제어 | 큐가 버퍼로 흡수 (강함) |
| 응답 지연(개별) | 낮음 (즉시 응답) | 접수는 빠르나 결과는 지연 |
| UX | 단순 (동기 결과) | 복잡 (추적/폴링) |
| 운영 복잡도 | 낮음 | 높음 (브로커 운영/순서/재처리) |
| 초과 트래픽 처리 | 429/503 + 재시도 안내 | 큐에 적재 후 순차 처리 |
| 장애 내구성 | Redis 의존, 영속성 약함 | 메시지 영속/재처리 강함 |
| 학습/어필 포인트 | 락 설계·경합 제어 | 스트림 처리·백프레셔·멱등성 |

---

### 1.3 권장 결정

> **주 아키텍처 = 안 A (Redis 동기 + 비가시 입장 제어).**
> **메시지 큐는 "예매 확정 후 후속 처리"에 부분 도입** (결제 결과 알림, 좌석 만료 이벤트, 통계 적재).

**근거**
1. KTX 예매의 본질 난제는 **"같은 좌석 동시 점유 제어"** → 분산 락으로 정면 돌파하는 게 학습/어필 가치가 가장 큼 (평가 기준 ②③).
2. 동기 응답이 예매 UX에 자연스럽고, 실제 KTX 앱처럼 **순번 비노출 입장 제어**가 도메인에 충실.
3. 안 B의 핵심 강점(스파이크 흡수)은 **활성자 수 제한 입장 제어 + 운행 조회 캐시**가 상당 부분 대체.
4. 그럼에도 **MQ를 "비동기 사이드"에 도입**해 메시지 큐 역량(멱등성, 재처리)도 포트폴리오에 포함.

**차별화 기회 (선택 / 강력 추천)**
> 핵심 예매 경로를 **안 A로 구현**하고, 동일 시나리오를 **안 B 방식으로도 PoC**하여
> *동일 부하에서 처리량/지연/정합성을 비교 측정* → "왜 A를 택했는가"를 **수치로 증명**.
> 이게 바로 영상이 말한 "다른 포트폴리오와 구분되는 나만의 관점".

### 1.4 확정 아키텍처 (논리도)
```
[Client]
  │ ⓪ 운행 리스트/매진 조회 (읽기 多, 약한 일관성)
  ▼
┌──────────┐ ──조회──▶ ┌──────────────────────────────┐
│  API 서버 │           │ Redis: 잔여 카운터 / TTL 캐시 │
└────┬─────┘           └──────────────────────────────┘
  │ ① 입장 토큰 요청
  ▼
┌─────────────────────────────┐
│ Redis: 활성자 카운터(상한 K) │ → 초과 시 429/503 + Retry-After (순번 X)
└──────────────┬──────────────┘
  │ ② 토큰 보유자만 동기 예매
  ▼
┌──────────┐   좌석 락    ┌──────────────────────────┐
│  예매 처리 │ ──────────▶ │ Redis: 좌석 분산 락        │
└────┬─────┘   임계영역   └──────────────────────────┘
     │ 상태전이(강한 일관성)
     ▼
┌──────────────────────────────┐      ┌────────────────────────┐
│ DB (SoT): Schedule /          │      │ Message Queue (비동기)  │
│ SeatInventory / Reservation   │ ───▶ │ 결제결과·만료·통계 이벤트 │
└──────────────────────────────┘      └────────────────────────┘
        ▲   │ 예매 확정 시 Redis 잔여 카운터 동기화
        │ HELD TTL 만료 스케줄러 → 미결제 좌석 자동 복구 → 카운터 복원
```

## Part 2. 검증 목표 — 측정 가능한 형태로

> 영상 핵심: **"기준을 설계하고 직접 측정하는 능력"**. 숫자 크기보다 *왜 그 기준인지 + Before/After*.

### 2.1 SLO (기준값 + 근거)
| # | 지표 | 목표 기준 | 근거 |
|---|------|-----------|------|
| S1 | 예매 API 응답 | **p95 ≤ 500ms, p99 ≤ 1s** | Google SRE 일반 웹 SLO 관례 |
| S2 | 운행 리스트/매진 조회 응답 | **p95 ≤ 200ms** | 읽기 트래픽 多, 모든 사용자의 진입점 |
| S2b | 매진/잔여석 표시 staleness | **≤ 2초** | 약한 일관성 허용 범위 |
| S3 | 처리량 | **동시 1,000 VUser에서 ≥ 200 TPS** | 명절 피크 가정 (조정 가능) |
| S4 | 정합성 (최우선) | **초과 판매 0건 / 중복 예매 0건** | 예매 시스템 절대 조건 |
| S5 | 가용성 | 부하 중 5xx 비율 **< 1%** (의도된 429/503은 제외) | 피크에도 서비스 유지 |
| S6 | 임계점 | **시스템이 무너지는 동시 사용자 수**를 숫자로 파악 | "어디서 무너지는가" |

> 기준값은 측정 후 현실에 맞게 보정한다. **중요한 건 "왜 이 값"을 설명할 수 있는 것.**

### 2.2 부하 테스트 시나리오
| ID | 시나리오 | VUser 프로파일 | 검증 대상 |
|----|----------|----------------|-----------|
| L1 | **단일 인기 좌석 동시 경쟁** | 1,000명이 동일 좌석 동시 요청 | S4 정합성(1명만 성공), 분산 락 |
| L2 | **정상 예매 플로우** | 0→1,000 점진 증가(ramp-up) | S1, S3 응답/처리량 |
| L2b | **자동 배정 처리량** | 1,000명이 `mode=AUTO`로 좌석 소진까지 | S3 처리량(경합 분산, L1 대비 비교) |
| L3 | **운행 조회 폭주(읽기 부하)** | 5,000명이 리스트/매진 동시 조회 | S2, 캐시/카운터 효과 |
| L4 | **입장 초과 부하** | 활성자 상한 K 초과로 요청 폭주 | 입장 제어(429/503) 정상 동작, S5 |
| L5 | **임계점 탐색(Stress)** | 한계까지 계속 증가 | S6 임계점 |
| L6 | **지속 부하(Soak)** | 중간 부하 30분 유지 | 메모리 누수/락 누수/HELD 만료 |

- 도구: **k6**(JS 시나리오) 또는 **nGrinder**(웹 UI). 점진적 부하 증가 필수.

### 2.3 정합성 검증 (자동화)
- [ ] 동시 N건 → DB `SOLD` 좌석 수 ≤ 총 좌석 수 (초과 0)
- [ ] 같은 좌석 `Reservation` CONFIRMED 행이 2개 이상 생기지 않음
- [ ] HELD 만료 후 좌석이 정확히 AVAILABLE로 복구
- [ ] 부하 종료 후 잔여 카운터(Redis)가 DB 실제 잔여로 수렴 (결과적 일관성)
- [ ] "매진 표시인데 예매 성공" 오탐 0건 (매진 표시는 보수적이어야 함)

### 2.4 캡처할 지표
- 응답시간 **p50 / p95 / p99**
- 처리량 **RPS / TPS**
- 에러율, **임계점**(무너지는 동시 사용자 수)
- 자원: CPU / 메모리 / DB 커넥션 / Redis 명령 지연

### 2.5 Before / After 실험 (포트폴리오의 하이라이트)
| 실험 | Before | After | 보여줄 것 |
|------|--------|-------|-----------|
| **E1 동시성 제어** | 락 없음 | 분산 락 적용 | 초과 판매 발생 → **0건** |
| **E2 입장 제어** | 제어 없음 | 활성자 수 제한 적용 | 5xx 폭증 → 429/503로 흡수, 코어 안정화 |
| **E3 조회 캐시** | 매 요청 DB 집계 | Redis 카운터/캐시 | 조회 p95·DB 부하 **대폭 감소** |
| **E4 (선택) A vs B** | Redis 동기 | MQ 비동기 | 처리량/지연/정합성 **수치 비교** |

> E1·E2·E3는 필수. 각 실험은 **그래프 + 한 줄 해석**으로 README/영상에 담는다.

### 2.6 합격 기준 (Definition of Done)
- [ ] S1~S5 SLO를 측정값으로 **충족 또는 미달 사유 설명**
- [ ] L1~L6 시나리오 결과 그래프 확보
- [ ] E1·E2·E3 Before/After 수치 확보
- [ ] 모든 SLO·기준값에 "왜 이 값인가" 근거 문서화
- [ ] 임계점(S6)을 숫자로 제시

## Part 3. 남은 결정 (설계 확정용)
1. **MQ 도입 범위** — (a) 비동기 사이드만 / (b) E3 비교 PoC까지 / (c) 도입 안 함
2. **좌석 선택 방식** — 사용자 직접 선택 vs 자동 배정 (직접 선택이 동시성 난이도·어필 ↑)
3. **HELD TTL** — 권장 5분
4. **활성 사용자 수 K** — 임계점(S6) 측정 후 역산하여 확정
5. **SLO 목표값** — 위 표값으로 시작 후 1차 측정으로 보정

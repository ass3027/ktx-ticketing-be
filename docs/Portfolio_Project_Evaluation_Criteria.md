## 1. 평가 영역 (4가지 방향성)

### ① 도메인 관심과 관찰력 — "왜 이 문제를 풀고 싶었는가"
- [ ] 직접 불편했던 경험에서 출발했는가 (가장 설득력 있는 동기)
- [ ] 문제 설정 자체가 도메인 이해를 증명하는가
- [ ] 단순 기술 연습이 아니라 실제 문제 인식에서 시작했는가

### ② 문제 설정과 해결 능력 — 문제 정의 → 설계 → 구현 → 검증
- [ ] 문제 정의가 명확한가
- [ ] AI에게 "무엇을 시킬지" 스스로 판단했는가
- [ ] 설계 결정마다 트레이드오프를 기록했는가

### ③ SW 기술과 공학적 접근 — 설계 의도 + 주도적 구현 + 측정
- [ ] 아키텍처를 설계하고 트레이드오프를 식별했는가
- [ ] AI를 주도적으로 활용해 구현했는가 (끌려가지 않았는가)
- [ ] 품질 기준·성능 지표를 설정하고 측정·해석했는가

### ④ 실행 가능한 결과물 — 면접관이 직접 확인 가능한 형태
- [ ] 배포 URL + GitHub 소스코드 (기본)
- [ ] README에 아키텍처·기술 선택 이유·트레이드오프 기록
- [ ] 동작 영상(YouTube) + 아키텍처 다이어그램
- [ ] (보안·정책상 URL 차단 시) 영상·문서로 보완

## 2. 성능 수치 만들기 — 3단계

> 실제 사용자가 없어도 된다. **"기준을 설계하고 직접 측정하는 능력"** 을 보여주는 게 핵심.
> 중요한 건 숫자의 크기가 아니라 **왜 그 기준을 잡았는지의 논리**. Before / After 차이가 있으면 충분.

### 1단계. 성능 지표 기준 설정
- 업계 레퍼런스를 참조해 내 서비스 기준을 직접 정의
- 참조 예: **Google SRE — SLO 설계법**
  - 예) "응답시간 p95 500ms 이하"
  - 예) "동시 사용자 100명 기준 처리량"

### 2단계. 부하 발생기로 부하 테스트
- 가상 사용자(VUser)를 직접 생성해 부하를 가함
- 점진적 부하 증가 시나리오 설계
- 대표 도구:

| 도구 | 특징 |
|------|------|
| **k6** | JavaScript로 시나리오 작성, 프론트·백 가리지 않고 시작 쉬움 |
| **nGrinder** | Naver 오픈소스(Apache 2.0, 무료), 웹 UI로 코드 없이 사용 가능 |
| JMeter | GUI 기반, 폭넓은 프로토콜 지원 |
| Gatling | 고성능, 코드 기반 시나리오 |

### 3단계. 응답시간 · 처리량 · 임계점 측정
- 시스템이 버틸 수 있는 한계를 숫자로 확인
- 측정 지표:
  - [ ] 응답시간 **p50 / p95 / p99**
  - [ ] 처리량 **RPS / TPS**
  - [ ] **임계점** — 어디서 무너지는가

## 3. 제출 전 셀프 체크리스트

- [ ] **왜 이 주제인지** 한 문장으로 설명할 수 있는가
- [ ] 내가 직접 설정한 **품질 기준과 측정 수치**가 있는가
- [ ] **배포 URL + GitHub 소스코드**는 기본, **README · 동작 영상**까지 갖췄는가
- [ ] **AI를 어떻게 활용**했고, **어디서 검증**했는지 설명할 수 있는가
- [ ] 핵심 코드·설계를 **구두로 설명**할 수 있는가
- [ ] 기술 선택에 **트레이드오프 근거**가 있는가
- [ ] 다른 포트폴리오와 **구분되는 나만의 관점**이 있는가

각 항목 0~5점. 합산으로 프로젝트 완성도/방향성 점검.

| 평가 영역 | 배점 | 점수 | 메모 |
|-----------|------|------|------|
| ① 도메인 관심·관찰력 | 5 | | |
| ② 문제 설정·해결 능력 | 5 | | |
| ③ SW 기술·공학적 접근 | 5 | | |
| ④ 실행 가능한 결과물 | 5 | | |
| 성능 측정 3단계 수행 | 5 | | |
| 셀프 체크리스트 충족 | 5 | | |
| **합계** | **30** | | |

> 권장: 시작 단계 15점 이상(방향 확정), 제출 단계 24점 이상 목표.

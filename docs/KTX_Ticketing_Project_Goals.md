| 항목 | 내용 |

## 1. 평가 영역 (4가지 방향성)

### ① 도메인 관심과 관찰력 — "왜 이 문제인가" (한 문장)
> "명절마다 KTX 예매창에서 좌석을 골라 결제 직전에 '이미 매진' 처리되거나, 새로고침 지옥을 겪으며 *왜 동시 예매가 이렇게 깨지는가*가 궁금했다."

- [x] 직접 겪은 불편(명절 예매 경험)에서 출발
- [x] 문제 설정이 도메인(티켓팅/동시성) 이해를 증명
- [x] 단순 CRUD 연습이 아니라 **동시성·정합성**이라는 실제 난제 인식

### ② 문제 설정과 해결 능력 — 정의 → 설계 → 구현 → 검증
- **문제 정의**: N명이 같은 좌석을 동시에 예매 시도할 때, 단 1명만 성공하고 좌석 재고가 정확히 차감되어야 한다. 매진 시 나머지는 대기열로 안내한다.
- **설계 결정 & 트레이드오프 (기록 대상)**:
  - 동시성 제어: DB 비관적 락 vs 낙관적 락 vs Redis 분산 락 → 선택 근거 기록
  - 대기열: Redis Sorted Set 기반 순번 vs 메시지 큐(Kafka) → 선택 근거 기록
  - 재고 차감: DB 트랜잭션 vs Redis 원자 연산(DECR) → 정합성 vs 성능 트레이드오프
- **AI 활용 판단**: 보일러플레이트/테스트 코드/부하 스크립트는 AI에 위임, 동시성 설계·락 전략은 직접 결정

### ③ SW 기술과 공학적 접근 — 설계 + 주도적 구현 + 측정
- **아키텍처(초안)**:
  - Client → API 서버(예매 요청) → 대기열(Redis) → 좌석 예매 처리(분산 락) → DB(좌석/예매)
- **핵심 구현 포인트**:
  - [ ] 분산 락으로 동일 좌석 동시 예매 차단 (Race Condition 해결 증명)
  - [ ] 좌석 재고 정합성: 동시 1,000건 요청 시 초과 판매 0건 검증
  - [ ] 대기열: 순번 부여, 입장 제어(Rate limiting), 이탈 처리
- **품질 기준**: 정합성(초과판매 0), 가용성, 응답시간 SLO 설정

### ④ 실행 가능한 결과물
- [ ] 배포 URL (또는 정책상 차단 시 동작 영상으로 보완)
- [ ] GitHub 소스코드
- [ ] README: 아키텍처 다이어그램 + 기술 선택 이유 + 트레이드오프 + Before/After 성능 수치
- [ ] 동작 영상(YouTube): 동시 예매 시연 + 부하 테스트 결과

## 2. 성능 목표 — 3단계 측정 계획

> "기준을 설계하고 직접 측정하는 능력"을 보여주는 게 핵심. 숫자 크기보다 **왜 그 기준인지 + Before/After**.

### 1단계. 성능 지표 기준 설정 (Google SRE SLO 참조)
| 지표 | 목표 기준 (예시 — 조정 가능) | 근거 |
|------|------------------------------|------|
| 응답시간 | 예매 API **p95 ≤ 500ms** | SRE 일반 웹 SLO 관례 |
| 처리량 | **동시 사용자 1,000명** 기준 처리 | 명절 피크 가정 |
| 정합성 | 초과 판매 **0건** | 예매 시스템 필수 조건 |

### 2단계. 부하 테스트
- 도구: **k6** (JS 시나리오) 또는 **nGrinder** (Naver 오픈소스, 웹 UI)
- 시나리오: VUser 100 → 500 → 1,000 → 임계점까지 **점진적 증가**
- 시나리오 종류: 단일 좌석 동시 경쟁 / 전체 예매 흐름 / 대기열 진입

### 3단계. 측정 & 해석
- [ ] 응답시간 **p50 / p95 / p99**
- [ ] 처리량 **RPS / TPS**
- [ ] **임계점** — 동시 몇 명에서 무너지는가
- [ ] **Before / After**: 락 적용 전(초과판매 발생) vs 후(0건), 튜닝 전후 처리량

## 3. 차별점 (나만의 관점)
- 단순 "예매 기능"이 아니라 **동시성 정합성 + 대기열 + 부하 측정**까지 공학적으로 증명
- 모든 설계 결정에 **트레이드오프 근거**를 README로 남김
- 실제 사용자 없이도 **부하 테스트로 성능을 수치화**

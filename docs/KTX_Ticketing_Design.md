# KTX 예매 시스템 — 설계

> 상위 문서: `KTX_Ticketing_Project_Goals.md`
> 범위: 도메인 모델 + 동시성 제어 + **입장 제어(비가시)** + **운행 조회·매진 표시** + API + 아키텍처
> 원칙: 모든 설계 결정에 **트레이드오프 근거**를 남긴다 (평가 기준 ②③).
> 정책 결정: 대기열 **순번을 노출하지 않는다(옵션 2)**. 실제 KTX 앱처럼 사용자는 줄을 보지 않고, 서버가 입장/부하를 제어한다.

## 1. 도메인 모델

### 1.1 핵심 엔티티
| 엔티티 | 설명 | 주요 속성 |
|--------|------|-----------|
| **Train (열차)** | KTX 운행 편 | id, 열차번호, 출발역, 도착역, 출발시각, 도착시각 |
| **Seat (좌석)** | 열차의 개별 좌석 | id, train_id, 호차, 좌석번호, 등급(일반/특실) |
| **SeatInventory (좌석 재고/상태)** | 특정 운행편의 좌석 예약 가능 여부 | id, schedule_id, seat_id, status(AVAILABLE/HELD/SOLD), version |
| **Schedule (운행 일정)** | 날짜별 운행편 | id, train_id, 운행일자, 총좌석수, 잔여좌석수, **매진여부(파생)** |
| **Reservation (예매)** | 사용자의 예매 건 | id, user_id, schedule_id, seat_id, status(HELD/CONFIRMED/CANCELLED), 생성시각, 만료시각 |
| **User (사용자)** | 예매자 | id, 이름, 이메일 |
| **EntryToken (입장 토큰)** | 비가시 입장 제어용 활성 세션 | token, user_id, 발급시각, TTL |

> 변경점: 순번을 노출하는 `WaitingQueueEntry`는 제거. 대신 **활성 세션 수만 제어**하는 `EntryToken`으로 대체(순번/대기번호 없음).

### 1.2 관계
- Train 1 — N Seat
- Train 1 — N Schedule (운행일자별)
- Schedule 1 — N SeatInventory (좌석별 상태)
- Reservation N — 1 SeatInventory (점유)
- User 1 — N Reservation

### 1.3 좌석 상태 전이 (핵심)
```
AVAILABLE ──(예매 시도 성공/임시 점유)──▶ HELD ──(결제 완료)──▶ SOLD
    ▲                                       │
    └──────(점유 만료 / 결제 취소)───────────┘
```
- **HELD**: 결제 대기 임시 점유 (예: 5분 TTL). 만료 시 자동 AVAILABLE 복귀.
- **매진(SOLD OUT)** = Schedule의 잔여좌석수 == 0 (AVAILABLE 좌석 없음).
- 이 상태 기계가 **동시성·정합성의 중심**.

## 2. 동시성 제어 전략 (프로젝트의 심장)

### 2.1 해결할 문제
> N명이 **같은 좌석**을 동시에 예매 시도 → 정확히 1명만 성공, 초과 판매 0건.

### 2.2 후보 비교 (트레이드오프)
| 전략 | 장점 | 단점 | 적합성 |
|------|------|------|--------|
| **DB 비관적 락** (`SELECT ... FOR UPDATE`) | 구현 단순, 강한 정합성 | 락 경합 시 처리량 급감, DB 부하 집중 | 좌석 단위 짧은 임계영역에 적합 |
| **DB 낙관적 락** (version 컬럼) | 락 없이 충돌만 감지, 읽기 성능 좋음 | 충돌 시 재시도 로직 필요, 경합 심하면 재시도 폭증 | 충돌 빈도 낮을 때 유리 |
| **Redis 분산 락** (Redisson/SETNX) | DB 부하 분산, 빠름, 다중 인스턴스 대응 | Redis 장애 대비 필요, 락 TTL 설계 주의 | **다중 서버 + 고경합에 권장** |
| **Redis 원자 연산** (DECR/Lua) | 재고 차감 자체가 원자적, 최고 처리량 | 좌석 "어떤 좌석" 매핑 별도 설계 필요 | 잔여수량 카운터에 적합 |

### 2.3 권장 설계 (조합)
1. **입장 토큰 보유자만** 예매 임계영역 진입 (동시 활성자 수 제한으로 경합 자체를 줄임)
2. 좌석 점유의 **원자적 선점**(아래 2.4) — 직접 선택/자동 배정 공통
3. 선점 성공 시 **DB 상태 전이**(AVAILABLE→HELD) + 낙관적 락(version)으로 2중 안전장치
4. 잔여좌석 카운터는 **Redis 원자 연산**으로 빠르게 노출, DB는 진실의 원천(SoT)

> 핵심 측정 포인트: 동시 1,000건 요청 → **초과 판매 0건** (Before/After로 제어 미적용 시 발생 vs 적용 시 0건 증명)

### 2.4 예매 모드 — 직접 선택 vs 자동 배정 (둘 다 지원)

두 모드는 **같은 좌석 재고를 공유**하므로, 단일한 **원자적 선점 지점**을 두어 충돌을 막는다.

#### 공유 자료구조: 가용 좌석 풀
- Redis Set `avail:{schedule_id}` = 현재 예매 가능한 seat_id 집합 (DB의 AVAILABLE 좌석과 동기화)
- **이 Set이 두 모드의 단일 선점 지점**. Redis 단일 스레드 특성으로 원자성 보장.

| 모드 | 선점 연산 | 동작 | 동시성 특성 |
|------|-----------|------|-------------|
| **직접 선택** | `SREM avail:{sched} {seat}` | 반환 1=내가 선점 / 0=이미 나감 | 인기 좌석에 **경합 집중** → 락/원자 선점의 진가 |
| **자동 배정** | `SPOP avail:{sched}` | 임의 가용 좌석 1개 원자적 획득 | 경합이 **분산** → 높은 처리량 |

- 두 연산 모두 **원자적**이라 "한 좌석을 둘이 가져가는" 상황이 원천 차단됨.
- 선점 성공자만 DB 상태전이(HELD) 수행 → DB는 SoT, Redis 풀은 빠른 선점 게이트.
- 취소/만료 시 `SADD avail:{sched} {seat}`로 풀에 복귀.

#### 분산 락과의 관계 (비교 실험 포인트)
- 직접 선택은 두 가지 구현이 가능 → **비교 대상**:
  - (가) **Redis Set 선점**(`SREM`) — 가볍고 빠름
  - (나) **좌석 단위 분산 락**(Redisson) — 직관적, DB 임계영역 보호에 명확
- 동일 부하에서 (가)/(나)의 처리량·정합성을 비교 측정 → "왜 이 방식인가"를 수치로 증명(실험 E1 확장).

> 결론: **선점은 Redis Set으로 통일**, 분산 락은 (a) DB 쓰기 임계영역 보호 또는 (b) 비교 실험용으로 활용.

## 3. 입장 제어 (Admission Control) — 비가시

### 3.1 목적
실제 KTX 앱처럼 **대기 순번을 보여주지 않고**, 서버가 동시 활성 사용자 수를 제한해 예매 로직과 DB를 보호한다.

### 3.2 설계
- **활성 세션 카운터**: Redis 카운터로 현재 "예매 진행 중" 사용자 수를 추적 (상한 K)
- **입장 토큰 발급**:
  - 활성자 < K → 즉시 입장 토큰 발급(TTL N분)
  - 활성자 ≥ K → `503 / 429` + `Retry-After`로 잠시 후 재시도 안내 (순번 노출 X)
- **예매 진입**: 입장 토큰 보유자만 예매 API 호출 가능
- **세션 해제**: 예매 완료/취소/토큰 TTL 만료 시 활성 카운터 감소
- **부하 셰딩**: 임계 초과 시 일부 요청을 빠르게 거절(load shedding)하여 전체 붕괴 방지

### 3.3 트레이드오프
| 방식 | 장점 | 단점 |
|------|------|------|
| **활성자 수 제한(token bucket류)** (채택) | 단순, 비가시, KTX 실제 동작에 충실, 시스템 보호 효과 명확 | "내 순번" 안내 불가(재시도 UX) |
| 보이는 대기열(Sorted Set 순번) | 사용자에게 진행감 제공 | KTX 도메인과 불일치(콘서트 티켓팅용), 부가 UI 필요 |

> README 트레이드오프 포인트: "왜 보이는 대기열이 아니라 비가시 입장 제어인가" = KTX 트래픽 특성 + 실제 앱 UX 충실성.

## 4. 운행 조회 & 매진 표시 (읽기 중심)

> 신규 요구사항: 예매 시 **현재 기차 운행 리스트**와 각 편의 **매진 여부**를 표시.

### 4.1 기능
- 출발역/도착역/날짜 조건으로 **운행편 리스트** 조회
- 각 편마다 상태 표시: **예매가능 / 잔여 N석 / 매진(SOLD OUT)**

### 4.2 핵심 난제 — "표시 정합성 vs 성능"
예매하려면 **모두가 먼저 리스트를 본다** → 읽기 트래픽이 쓰기(예매)보다 훨씬 많다. 매 조회마다 DB `COUNT`를 하면 피크에 DB가 먼저 죽는다.

| 전략 | 정합성 | 성능 | 메모 |
|------|--------|------|------|
| 매 요청 DB 집계 | 정확 | 나쁨(고부하 위험) | 비채택 |
| **Redis 잔여 카운터 조회** (채택) | 결과적 일관성(약간 stale 허용) | 좋음 | 예매 시 카운터 갱신, 표시는 best-effort |
| 짧은 TTL 캐시 | 약간 stale | 매우 좋음 | 카운터와 병행 가능 |

### 4.3 설계 원칙 (중요)
- **표시(매진 여부/잔여석)는 결과적 일관성 허용**: "잔여 1석"을 보고 들어갔는데 매진일 수 있음 → **정상**. 표시는 안내일 뿐, **최종 판정은 예매 임계영역(강한 일관성)에서** 한다.
- 따라서 **두 단계의 일관성 모델을 분리**한다:
  - 조회/표시 = 빠름·약한 일관성 (Redis 카운터/캐시)
  - 예매 확정 = 강한 일관성 (분산 락 + DB 상태전이)
- 오탐 최소화: 매진은 비교적 정확히(잔여 0이면 즉시 매진 표시), "가능"은 best-effort.

> 이 일관성 분리 자체가 강한 어필 포인트(평가 기준 ②③, "트레이드오프 근거").

## 5. API 설계 (초안)

| 메서드 | 경로 | 설명 |
|--------|------|------|
| GET | `/api/schedules?dep=서울&arr=부산&date=...` | **운행편 리스트 + 매진/잔여석 표시** |
| GET | `/api/schedules/{id}/seats` | 좌석 현황(개별 좌석 AVAILABLE/SOLD) 조회 |
| POST | `/api/entry` | 입장 토큰 요청 (활성자 초과 시 429/503 + Retry-After) |
| POST | `/api/reservations` | 좌석 예매(HELD) — 입장 토큰 필요. body: `mode=SEAT`(seat_id 지정) 또는 `mode=AUTO`(자동 배정) |
| POST | `/api/reservations/{id}/confirm` | 결제 확정(SOLD) |
| DELETE | `/api/reservations/{id}` | 예매 취소 |

## 6. 아키텍처 (논리 구성)

```
[Client]
   │  ① 운행 리스트/매진 조회 (읽기 多)
   ▼
[API 서버] ──조회──▶ [Redis: 잔여 카운터 / 짧은 TTL 캐시]  (약한 일관성·빠름)
   │  ② 입장 토큰 요청
   ▼
[입장 제어] ──▶ [Redis: 활성자 카운터(상한 K)]  → 초과 시 429/503
   │  ③ 토큰 보유자만 예매(강한 일관성)
   ▼
[예매 처리] ──▶ [Redis: 좌석 분산 락]  ── 임계영역: 상태 전이 ──┐
   │                                                          │
   ▼                                                          ▼
[DB: Schedule / SeatInventory / Reservation]  ← 진실의 원천(SoT)
   │   (예매 확정 시 Redis 잔여 카운터 동기화)
   ▼
[HELD TTL 만료 스케줄러] → 미결제 좌석 자동 복구 → 카운터 복원
   │
   └──▶ [Message Queue] (비동기 사이드: 결제결과·만료·통계 이벤트)
```

## 7. 검증·측정 연결 (목표 문서와 매핑)
- **정합성**: 동시 1,000건 → 초과 판매 0건 (통합 테스트 + 부하 테스트)
- **예매 응답**: p95 ≤ 500ms
- **조회 응답**: 운행 리스트 p95 ≤ 200ms (읽기 부하 큰 경로)
- **처리량/임계점**: 동시 1,000명 기준 RPS/TPS, 무너지는 지점
- **Before/After**: ①락 미적용 vs 적용(초과판매), ②입장 제어 미적용 vs 적용(안정성), ③조회 DB직접 vs 캐시(부하)

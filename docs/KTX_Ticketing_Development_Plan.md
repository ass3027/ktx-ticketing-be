## 1. 페이즈별 상세

### P0. 프로젝트 셋업 (0.5일)
- [ ] Git 레포 + 브랜치 전략, 디렉터리 구조
- [ ] 기술 스택 확정 (Redis 확정, 그 외 결정)
- [ ] 로컬 Docker Compose (앱/DB/Redis)
- [ ] 기본 CI(빌드+테스트) 파이프라인
- **DoD**: `docker compose up`으로 빈 앱+DB+Redis 기동

### P1. 설계 확정 (1.5일)
- [ ] **미정값 결정**: HELD TTL(5분?), 결제 2단계 여부, MQ 도입 범위
- [ ] ERD 확정 + DDL 작성 (Train/Seat/Schedule/SeatInventory/Reservation/User)
- [ ] Redis 키 설계: `avail:{sched}`(Set), 잔여 카운터, 활성자 카운터, EntryToken
- [ ] seed 스크립트(테스트 데이터: 50편×1,000석, user 1만)
- **DoD = M1 설계 동결**: ERD/DDL/Redis 키/seed 확정

### P2. 핵심 PoC — 리스크 First (3일)
> 가장 위험한 "동시 선점 정합성"을 최소 코드로 먼저 증명.
- [ ] 좌석 선점 PoC: `SREM avail`(직접) / `SPOP avail`(자동)
- [ ] DB 상태전이(AVAILABLE→HELD) + 낙관 락(version)
- [ ] (비교용) 분산 락(Redisson) 버전 PoC
- [ ] 동시 1,000요청 테스트 → **초과 판매 0건 확인**
- **DoD = M2**: 동시성 PoC가 초과 0건. 아키텍처 확신 확보(실패 시 설계 회귀)

### P3. 기능 구현 (5일)
**① 조회/입장 (2일)**
- [ ] 운행 리스트 조회 + 매진/잔여석 표시 (Redis 카운터/캐시, 약한 일관성)
- [ ] 입장 제어: 활성자 카운터(상한 K), 초과 시 429/503 + Retry-After, EntryToken 발급/만료

**② 예매/결제 (3일)**
- [ ] 예매 API: `mode=SEAT` / `mode=AUTO` (P2 선점 로직 통합)
- [ ] 결제 확정(HELD→SOLD), 예매 취소, 카운터/활성자 동기화
- [ ] HELD TTL 만료 스케줄러 → 좌석/카운터/`avail` 복구
- [ ] Redis-DB reconcile 잡: 카운터/`avail` 드리프트 주기 보정 → DB(SoT)로 수렴 (리스크 §4 대응)
- [ ] 통합 테스트(정합성 자동화): 중복/초과/만료복구 + reconcile 수렴
- **DoD = M3**: 정상 흐름 16단계 E2E 동작 + 예외 흐름 6종 처리

### P4. 성능 측정 (4일)
> `KTX_Ticketing_Performance_Test.md` 그대로 실행.
- [ ] 부하 환경 구축(k6/nGrinder) + 서버 모니터링(CPU/메모리/DB/Redis)
- [ ] L1 정합성 → L2/L2b 처리량 → L3 조회 → L4 입장초과 → **L5 임계점(→K 역산)** → L6 soak
- [ ] 실험 **E1**(선점/락) · **E2**(입장 제어) · **E3**(조회 캐시) Before/After
- [ ] 결과 기록표 + 그래프 캡처
- **DoD = M4**: SLO 충족/미달 사유 + Before/After 그래프 + 임계점 수치 확보

### P5. 비동기 사이드 (2일)
- [ ] MQ 연동: 예매확정/취소/만료 이벤트 발행
- [ ] 컨슈머: 알림(mock)/통계 적재, **멱등성** 처리
- [ ] (선택) 실험 E4: Redis 동기 vs MQ 비동기 비교 PoC
- **DoD**: 이벤트 발행/소비 동작, 멱등성 테스트 통과

### P6. 산출물 & 마무리 (3일)
> 평가 기준 ④ "면접관이 직접 확인 가능한 형태".
- [ ] README: 문제정의·아키텍처 다이어그램·기술선택 이유·**트레이드오프**·성능 그래프
- [ ] 아키텍처/시퀀스 다이어그램 이미지화
- [ ] 배포(URL) — 차단 시 동작 영상으로 보완
- [ ] 동작 영상(YouTube): 정상 흐름 + 동시성 시연 + 부하 결과
- [ ] **셀프 체크리스트 7항목** 최종 점검(`Portfolio_Project_Evaluation_Criteria.md`)
- **DoD = M5 제출 가능**

## 5. 범위 분리 (MoSCoW)
- **Must**: 조회·매진, 입장 제어, 예매(SEAT/AUTO), 결제/취소, 만료, 정합성, L1·L2·L5, E1·E2
- **Should**: L3·L4·L6, E3, 배포, 동작 영상
- **Could**: MQ 비동기 사이드, E4(A vs B) 비교
- **Won't(이번 범위 X)**: 실제 PG 연동, 실 사용자 트래픽, 다국어/모바일앱

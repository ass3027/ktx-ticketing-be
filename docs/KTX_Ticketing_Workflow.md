# KTX 예매 시스템 — 전체 동작 흐름 시나리오

> 상위 문서: `KTX_Ticketing_Design.md`, `KTX_Ticketing_Architecture_and_Verification_Goals.md`
> 목적: 사용자 여정을 단계별 시퀀스로 정리. 정상 흐름 + 예외/동시성 흐름 포함.

---

## 0. 등장 요소
| 약칭 | 구성요소 |
|------|----------|
| **U** | 사용자(Client) |
| **API** | API 서버 (예매/조회/입장) |
| **RC** | Redis: 잔여 카운터 / 조회 캐시 (약한 일관성) |
| **RA** | Redis: 활성자 카운터 (입장 제어, 상한 K) |
| **RP** | Redis: 가용 좌석 풀 `avail:{schedule_id}` (Set) — 두 모드 공통 선점 지점 |
| **RL** | Redis: 좌석 분산 락 (DB 임계영역 보호 / 비교 실험용) |
| **DB** | RDB (Schedule/SeatInventory/Reservation) — 진실의 원천(SoT) |
| **SCH** | HELD TTL 만료 스케줄러 |
| **MQ** | 메시지 큐 (비동기 사이드: 알림/통계/만료 이벤트) |

좌석 상태: `AVAILABLE → HELD → SOLD` (만료/취소 시 AVAILABLE 복귀)

## 1. 정상 흐름 (Happy Path)

### 단계별
| # | 단계 | 행위 | 비고 |
|---|------|------|------|
| 1 | **운행 조회** | U → API: 서울→부산, 6/30 검색 | |
| 2 | | API → RC: 각 편 잔여수/매진 조회 | 캐시/카운터(빠름, 약한 일관성) |
| 3 | | API → U: 운행 리스트 + `예매가능/잔여 N석/매진` 표시 | staleness ≤ 2초 허용 |
| 4 | **입장 요청** | U → API → RA: 활성자 < K? | |
| 5 | | RA → API: 입장 허용 → **EntryToken 발급**(TTL) | 활성자 카운트 +1 |
| 6 | **예매 요청** | U(토큰 보유) → API: `mode=SEAT(12A)` 또는 `mode=AUTO` | 토큰 없으면 거절 |
| 7 | | **선점**: SEAT → `SREM avail {12A}` (1이면 성공) / AUTO → `SPOP avail` (임의 좌석 획득) | RP에서 원자적 선점 |
| 8 | | API → DB: 선점한 좌석 `AVAILABLE→HELD`(낙관 락 version 체크) | 강한 일관성 |
| 9 | | API → RC: 잔여 카운터 -1 동기화 | 표시 갱신 |
| 10 | | (분산 락 사용 시) DB 임계영역만 `lock/unlock` | 선택적 |
| 11 | | API → U: **예매 성공(HELD)** + 결제 마감시각(예: +5분) | |
| 12 | **결제 확정** | U → API: 결제 확정 | (PG는 Mock 가능) |
| 13 | | API → DB: 좌석 `HELD→SOLD`, Reservation `CONFIRMED` | |
| 14 | | API → RA: 활성자 카운트 -1 (세션 종료) | |
| 15 | | API → MQ: `예매확정` 이벤트 발행 | 알림/통계 비동기 |
| 16 | | API → U: **예매 완료** | |

### 시퀀스 다이어그램
```
U        API        RC/RA/RL        DB           MQ
│ 조회    │           │              │            │
│────────▶│──RC 조회──▶│              │            │
│◀────────│  리스트+매진             │            │
│ 입장요청 │           │              │            │
│────────▶│──RA 확인──▶│ (활성<K)     │            │
│◀─토큰────│  +1        │              │            │
│ 예매     │           │              │            │
│────────▶│──RL lock──▶│              │            │
│         │──────상태전이 HELD───────▶│            │
│         │──RC -1────▶│              │            │
│         │──RL unlock▶│              │            │
│◀─성공────│ (HELD, 5분 마감)         │            │
│ 결제확정 │           │              │            │
│────────▶│────────HELD→SOLD────────▶│            │
│         │──RA -1────▶│              │            │
│         │────────예매확정 이벤트──────────────────▶│
│◀─완료────│           │              │            │
```

## 2. 핵심 동시성 흐름 (프로젝트의 심장)

### 2.1 직접 선택 — 같은 좌석 1,000명 경쟁
```
U1..U1000  동시에 mode=SEAT(12A) 요청 (모두 토큰 보유)
        ▼
   [RP] SREM avail {12A}  ── 단 1명(U_k)만 1 반환(선점), 나머지는 0
        ▼
   U_k: DB AVAILABLE→HELD 성공 → 잔여 -1
        ▼
   나머지 999명: SREM=0 → "이미 선택된 좌석입니다" 응답
```
- **경합 집중형** — 인기 좌석에 부하 몰림. 분산 락 vs SREM 선점 **비교 대상**(실험 E1 확장).

### 2.2 자동 배정 — 1,000명이 "아무 좌석이나" 요청
```
U1..U1000  동시에 mode=AUTO 요청 (가용 좌석 800석 가정)
        ▼
   [RP] SPOP avail  ── 각자 서로 다른 좌석을 원자적으로 획득
        ▼
   앞선 800명: 좌석 획득 → DB HELD → 잔여 -1
   나머지 200명: SPOP nil → "매진" 응답
```
- **경합 분산형** — 좌석이 고루 분산되어 처리량↑. 동일 좌석 충돌 없음.

### 2.3 공통 결과 (검증)
- 두 모드 모두 선점이 **원자적**이라: CONFIRMED ≤ 가용 좌석 수, 초과 판매 **0건** (S4)
- 두 모드가 **동시에 같은 schedule에 섞여 들어와도** 같은 `avail` Set을 공유 → 한 좌석을 둘이 가져가는 일 없음
- **측정**: 제어 미적용 시 초과 발생 → 적용 시 0건 (E1). 직접 선택 vs 자동 배정의 **처리량/지연 차이**도 비교 그래프로 제시

## 3. 예외 흐름

### 3.1 매진 좌석 조회/시도
- 조회 시 잔여 0 → **`매진(SOLD OUT)`** 표시 (보수적: 매진 오탐 0)
- (stale로) 가능처럼 보고 예매 시도 → 7~8단계에서 **최종 판정 실패** → "매진되었습니다" 응답 → **정상 동작**

### 3.2 입장 초과 (활성자 ≥ K)
- 4단계에서 RA가 한도 초과 판단 → API → U: **`429/503 + Retry-After`** (순번 없음)
- U는 잠시 후 재시도 → 자리 나면 입장 (실험 E2)

### 3.3 결제 미완료 → HELD 만료
- 11단계 후 사용자가 5분 내 결제 안 함
- SCH가 만료된 HELD 감지 → 좌석 `HELD→AVAILABLE` 복구 → RC 잔여 +1 → RA 활성자 -1
- MQ에 `좌석만료` 이벤트 발행(통계/모니터링)
- → 좌석이 다시 예매 가능해짐

### 3.4 사용자 예매 취소
- U → API: 취소 → DB `HELD/CONFIRMED → CANCELLED`, 좌석 AVAILABLE 복구 → RC +1, RA -1
- MQ `예매취소` 이벤트

### 3.5 토큰 없이 예매 시도
- 6단계에서 토큰 누락/만료 → API: **거절(401/403)** → 입장 요청부터 다시

### 3.6 Redis 카운터 ≠ DB (정합성 드리프트)
- 표시 카운터는 약한 일관성이라 일시적 오차 가능
- 보정: 주기적 재동기화(reconcile) 잡 + 부하 종료 후 **카운터가 DB로 수렴** (검증 2.3)

## 4. 상태/일관성 요약
| 구간 | 일관성 | 근거 |
|------|--------|------|
| 운행 조회·매진 표시 | 약함(stale ≤ 2초) | 읽기 부하↑, 표시는 안내 |
| 입장 제어 | 근사(활성자 수 추정) | 보호가 목적, 정밀도 덜 중요 |
| 좌석 예매(상태전이) | **강함** | 초과 판매 0건 절대 조건 |

---

## 5. 흐름이 검증 목표와 맺는 연결
- 1번 정상 흐름 → S1(예매 p95), S2(조회 p95)
- 2번 동시성 흐름 → S4(정합성), 실험 E1
- 3.2 입장 초과 → S5(가용성), 실험 E2
- 조회 경로 → 실험 E3(캐시 효과)
- 3.3 만료 → Soak 테스트(L6)에서 누수 없이 복구되는지 확인

## 결정됨 / 미정
- ✅ **좌석 선택 방식** — 직접 선택(`mode=SEAT`) + 자동 배정(`mode=AUTO`) **둘 다 지원**. 공유 `avail` Set으로 공존
- ⬜ **결제 단계 존재 여부** — HELD→SOLD 2단계 유지 vs 즉시 SOLD 단순화
